開発者や AI スタートアップのメンバーなら、こんな悩みがあるはずです：ブログで情報を発信し、ブランドを構築し、オーガニックトラフィックを増やしたい。でもブログの構築に、もう一つのプロダクトを作るような労力はかけたくない。

それが CitePo が存在する理由です。

## 既存ソリューションの問題点

### WordPress / Ghost

高機能な CMS プラットフォームですが、負担も大きいです：

- **ホスティングの手間** — サーバー、データベース、継続的なメンテナンスが必要
- **セキュリティの懸念** — 頻繁なアップデートと脆弱性パッチ
- **大半のブログにはオーバースペック** — Markdown を公開したいだけなのに CMS を管理する必要がある

<Callout type="warning">
従来の CMS は継続的なサーバーメンテナンスとセキュリティパッチが必要です。その労力はコンテンツの執筆に使うべきです。
</Callout>

### 静的サイトジェネレーター（Hugo、Jekyll、Gatsby）

開発者が求めるものに近いですが、まだ摩擦があります：

- **複雑な設定** — テーマ、プラグイン、ビルドパイプライン
- **重い依存関係** — 数百のパッケージ、頻繁な破壊
- **リポジトリの90%がエンジン** — コンテンツがフレームワークコードに埋もれる

<Callout type="warning">
SSG では、コンテンツに集中するのではなく、複雑なビルドシステムのメンテナンスに追われてしまいます。
</Callout>

### Medium / Substack

始めるのは簡単ですが：

- **所有権がない** — コンテンツは他社のプラットフォーム上にある
- **ブランディングの制限** — プラットフォーム上の他のブログと同じ見た目
- **SEO の制限** — 自分のドメインではなく、プラットフォームのドメイン権威を構築している

<Callout type="warning">
ホスト型プラットフォームは利便性とコントロールを交換します。コンテンツ、ブランド、SEO はすべて他者のものになります。
</Callout>

## クイックスタート

ブログの作成はわずか数ステップ：

<Steps>
  <Step title="ブログを作成">
    スキャフォールドコマンドで新しいブログプロジェクトを生成：
    ```bash
    npx citepo new my-blog
    ```
  </Step>
  <Step title="プロジェクトディレクトリに移動">
    ```bash
    cd my-blog
    ```
  </Step>
  <Step title="開発サーバーを起動">
    ブログをローカルでプレビュー：
    ```bash
    npx citepo dev
    ```
  </Step>
</Steps>

以上です。アカウント登録不要、クレジットカード不要、設定ウィザード不要。すぐに書き始められます。
